## 1) ما هو التطبيق ولماذا

تطبيق ويب عربي (يمين‑لليسار) لإدارة **مكتب استشارات هندسي** في الدمام، يحلّ محلّ المتابعة الذهنية والورقية بمساحة عمل واحدة مشتركة:
- **المدير العام** يوزّع العمل ويمنح الصلاحيات ويرى كل شيء (بما فيه المال).
- **المهندس** يرى مشاريعه ومهامه، يحدّث نسبة الإنجاز، يضيف ملاحظات ومرفقات (وملاحظات صوتية) — **دون أن يرى أي مبالغ مالية إطلاقًا**.
- **المحاسب** يدير العملاء والعروض والفواتير والتحصيل والتقارير المالية — **دون إدارة الفريق أو الصلاحيات**.

البيانات مشتركة لحظيًا بين كل الأجهزة (Realtime)، لا تعتمد على تخزين المتصفح. التطبيق قابل للتثبيت على الهاتف كتطبيق (PWA).

## 5) الوحدات والميزات بالتفصيل

- **لوحة التحكم:** مؤشرات حسب الدور — للمدير/المحاسب أرقام مالية (المتبقّي، المحصّل، المتأخر)، وللمهندس مهامه المفتوحة/المتأخرة/العاجلة. روابط سريعة لكل قسم.
- **المشاريع:** الحالة (تخطيط/قيد التنفيذ/متوقف/مكتمل/ملغى)، نسبة الإنجاز، تواريخ بدء/استحقاق حقيقية، العميل المرتبط، وأعضاء المشروع. الميزانية تظهر للمدير/المحاسب فقط.
- **المهام:** ملكية واضحة دومًا (المسند إليه + المشروع)، أولوية، حالات بآلة حالات مفروضة بقاعدة البيانات (لا «حالات مجهولة»)، تحديث نسبة الإنجاز، إرسال للمراجعة، ملاحظات، معالم، **مرفقات وملاحظات صوتية**، وسجلّ كامل لدورة الحياة. تبويبات: الكل/مهامي/بانتظار المراجعة/متأخرة/عاجلة/مكتملة.
- **العملاء:** أفراد وشركات، بيانات تواصل بروابط مباشرة (اتصال/واتساب/بريد)، وهوية ضريبية اختيارية (رقم ضريبي + سجل تجاري) للعملاء المؤسسيين.
- **العروض:** بكل الحالات (مسودة/مُرسل/مقبول→يتحوّل لمشروع/مرفوض/منتهٍ).
- **الفواتير والدفعات:** إصدار فاتورة (مجموع فرعي + ضريبة ١٥٪ + إجمالي)، تسجيل دفعات (كلية/جزئية) بطرق دفع، **سند قبض** قابل للطباعة لكل دفعة، عكس دفعة، إلغاء/تصفير، و**قائمة تحصيل** للمتأخرات مرتّبة بالأقدم مع متابعة بنقرة.
- **التقارير المالية:** المتبقّي، المحصّل، المتأخر بفئات تقادم، فواتير الشهر، وغيرها.
- **الفريق:** إنشاء حسابات الموظفين، تحديد الأدوار، تفعيل/تعطيل، إعادة تعيين كلمة المرور.
- **الصلاحيات:** منح المهندس صلاحيات إضافية بدقّة (مع بقاء حظر المال).
- **معرض الأعمال:** واجهة المكتب أمام عملائه الجدد (أعمال منجزة بصور).
- **التقويم:** مواعيد استحقاق المهام/المشاريع/الفواتير في شهر واحد، مع تمييز المتأخرات.
- **الإشعارات:** جرس داخل التطبيق + **تذكيرات يومية تلقائية** (فواتير متأخرة للمدير/المحاسب، مهام مستحقة للمهندس، عروض منتهية للمدير) مع حفظ عزل المالية.
- **إعدادات المكتب:** الهوية الرسمية (الاسم، السجل التجاري، الرقم الضريبي، العنوان، الهاتف، تذييل الفاتورة) — تُفعّل الفاتورة الضريبية ورمز ZATCA تلقائيًا.
- **النسخ الاحتياطي/التصدير:** تصدير البيانات قبل إدخال بيانات حقيقية.

## 6) بداية سريعة لكل دور

**المدير:** سجّل الدخول → أدخل هوية المكتب في «إعدادات المكتب» → أضف موظفيك من «الفريق» → أنشئ عميلًا فمشروعًا فمهمة → أسند المهمة لمهندس. للمالية: أنشئ عرضًا/فاتورة وتابع التحصيل.

**المهندس:** سجّل الدخول → «مهامي» → افتح مهمة → حدّث نسبة الإنجاز / أضف ملاحظة أو مرفقًا/تسجيلًا صوتيًا → «إرسال للمراجعة» عند الانتهاء. (لن ترى أي مبالغ.)

**المحاسب:** سجّل الدخول → «الفواتير» لإصدار فاتورة وتسجيل دفعة وطباعة سند قبض → «التحصيل» لمتابعة المتأخرات → «التقارير» للملخّص المالي.

## 7) عزل المالية والأمان

- **الجوهرة:** المهندس لا يرى أي مبلغ — لا في الواجهة (لا عمود ميزانية، لا فواتير/تقارير/عروض، والوصول المباشر لأي صفحة مالية يُرَدّ بـ«لا تملك صلاحية الوصول») ولا في قاعدة البيانات (سياسات RLS تُعيد **صفر صفوف** لرمز المهندس).
- الصلاحيات مفروضة على **طبقة قاعدة البيانات** (RLS + دوال SECURITY DEFINER تتحقق داخليًا)، لا بإخفاء الواجهة فقط.
- لا أسرار في الكود أو Git. مفتاح الخدمة لا يُستخدم في كود المتصفح إطلاقًا.
- سياسة محتوى صارمة (CSP) تحجب `eval` والنصوص غير الموثوقة.

## 8) الفواتير الضريبية ورمز ZATCA

عند إدخال هوية المكتب الضريبية، تتحوّل الفاتورة تلقائيًا إلى **«فاتورة ضريبية مبسطة»** (أو «فاتورة ضريبية» كاملة إذا كان للعميل رقم ضريبي) مع:
- مجموع فرعي + **ضريبة قيمة مضافة ١٥٪** + إجمالي (الحساب دقيق ومُتحقَّق منه).
- **رمز ZATCA QR** (المرحلة الأولى) مُولَّد تلقائيًا على الفاتورة القابلة للطباعة.
- **سند قبض** احترافي لكل دفعة يحمل بيانات المكتب الضريبية والمبلغ والمتبقّي.

## 9) تطبيق الجوال (PWA) والعمل دون اتصال

- قابل للتثبيت على الهاتف (الوضع standalone، يمين‑لليسار، أيقونات كاملة).
- يعمل دون اتصال بشكل أساسي (service worker + صفحة `offline`).
- التصميم مستجيب فعلًا: الجداول العريضة تتحوّل إلى **بطاقات** على الجوال، مع شريط تنقّل سفلي.

## 10) ما أُنجز واختُبر في الجلسات الأخيرة

**نُشر للإنتاج (يدويًا — لا يوجد نشر تلقائي عند الدفع على هذا المشروع):**
- `must()` يُظهر فشل القراءات الحرجة كشاشة خطأ بدل قائمة فارغة صامتة، + تحسينات إتاحة وتواريخ.
- معلم `<main>` لصفحات الدخول (رفع درجة الإتاحة إلى **١٠٠**) + `robots.txt` خاص يمنع فهرسة التطبيق الخاص.

**اختبار شامل (مُتحقَّق حيًّا — أغلبه على العرض الغني، مع فحص نقطي على الإنتاج):**
- **عزل الأدوار الثلاثة:** المهندس بلا مال، المحاسب يرى المال ويُمنع من الإدارة، المدير كامل.
- **صحّة المالية:** ضريبة ١٥٪ دقيقة، رمز ZATCA موجود فعلًا، الدفعات الجزئية والمتبقّي صحيحة، سند القبض كامل.
- **التحقق المنتجي:** رفض تاريخ بدء بعد الاستحقاق برسالة عربية ودودة (ثلاث طبقات: عميل/خادم/قاعدة بيانات)، وملكية مهام واضحة بلا حالات مجهولة.
- **الإتاحة:** ١٠٠ على صفحة الدخول وعلى صفحة مُصادَقة غنية.
- **الأداء:** سرعة عرض جيدة (LCP ١.٤٣ث، CLS ٠) ومنطقة خادم مُتجاورة مع قاعدة البيانات (فرانكفورت).
- **PWA:** كل الأصول تُخدَم بنجاح؛ قابل للتثبيت ويعمل دون اتصال.

**النتيجة:** لا عيوب موضوعية جديدة عدا بندَي الإتاحة/SEO اللذين أُصلحا ونُشرا. التطبيق ناضج وجاهز للإنتاج.

## 11) التقنيات والبنية التحتية

- **الواجهة:** Next.js (App Router) + TypeScript + Tailwind + shadcn/Radix.
- **الخلفية:** Supabase (مصادقة، Postgres، تخزين، Realtime، RLS).
- **النشر:** Vercel (منطقة `fra1` فرانكفورت، مُتجاورة مع قاعدة البيانات في eu-central-1).
- مشروعان منفصلان تمامًا للإنتاج والعرض (قاعدة بيانات + نشر لكلٍّ).
- جودة: فحص الأنواع + التدقيق + اختبارات الوحدة + إثبات RLS + Playwright — كلها خضراء.

## 13) الجاهزية للإنتاج وبوابة ما قبل الإطلاق

التطبيق جاهز للمراجعة والاستخدام الآن. قبل **الإطلاق الرسمي ببيانات حقيقية حسّاسة**، يُنصح بـ:
1. **ترقية Supabase إلى Pro** (نسخ احتياطي تلقائي + يتيح تفعيل البند التالي).
2. **تفعيل حماية كلمات المرور المسرّبة (HIBP)** — متاحة على خطة Pro فأعلى (المؤسسة حاليًا على المجانية). المسار: Authentication → Providers → Email. (التطبيق أصلًا يفرض حدًّا أدنى ١٢ حرفًا + تغيير إجباري أول دخول.)
3. إدخال هوية المكتب في «إعدادات المكتب».
4. مراقبة حدود الخطة المجانية (تخزين ١GB، قاعدة ٥٠٠MB) قبل التوسّع.

## 14) ما يتكشّف بالاستخدام الفعلي

تسمية أدوار الموظفين الدقيقة، مراحل المشاريع الفعلية، الرقم الضريبي الحقيقي، وتفضيلات صياغة بعض المصطلحات — كلها تتكشّف بالاستخدام، والتطبيق مهيّأ لاستيعابها دون تعديل برمجي.

## 15) الخطوات التالية المقترحة

- مشاركة رابط العرض مع حمزة لتجربته على مهله بكل دور.
- جمع ملاحظاته على المصطلحات والصياغة وتطبيقها.
- عند قرار الإطلاق: تنفيذ بوابة ما قبل الإطلاق (القسم 13).
- تحسينات اختيارية مستقبلية: هياكل تحميل لكل صفحة، تنبيهات أغنى، وتقارير إضافية حسب الحاجة.
